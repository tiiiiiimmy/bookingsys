## 4. 已知后端缺陷（TC-RS-10，xfail 保留）

`POST /admin/reschedule-requests/{id}/approve` 的实现 `ApproveRescheduleRequestAsync` 用**不加锁的普通 `SELECT`** 读取申请状态（无 `FOR UPDATE`）。在 InnoDB REPEATABLE READ 下，对**同一条 pending 申请**的两次并发批准会各自读到 `pending`、同时通过"已复核"校验并提交，因此 **exactly-one-wins 仅在串行场景下成立**。

- 实测：两次并发批准均返回 OK（`Received length: 2`，期望 `1`）。该失败**确定、可复现，非 flaky**。
- 处理：保留断言为正确的预期行为，场景打 `@fail`（xfail）并在 feature 内注释根因；待后端把该读取改为加锁读后移除标记（届时会"意外通过"以提醒摘除）。
