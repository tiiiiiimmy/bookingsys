## 4. Known Backend Defect (TC-RS-10, xfail retained)

`POST /admin/reschedule-requests/{id}/approve` → `ApproveRescheduleRequestAsync` reads request status with an **unlocked plain `SELECT`** (no `FOR UPDATE`). Under InnoDB REPEATABLE READ, two concurrent approvals of the same pending request can both read `pending`, pass the "already reviewed" check, and commit — **exactly-one-wins holds only under serial execution**.

- Observed: both concurrent approvals return OK (`Received length: 2`, expected `1`). Failure is **deterministic, reproducible, not flaky**.
- Handling: keep assertion as correct expected behavior; tag `@fail` (xfail) with root-cause comment; remove tag after backend adds locking (will "unexpectedly pass" as a reminder).
