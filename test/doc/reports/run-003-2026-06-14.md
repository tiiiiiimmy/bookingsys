## 4. 调查：Run #002 之后的偶发失败根因

在 #002（7/7 绿）与本次之间，曾出现 `Booking › …payment succeeds` 与 `Reschedule …` 偶发失败，现象统一为：**预约流程后半段页面变空白（SPA 未渲染），用例等元素超时**。逐一排除：

- **非数据污染**：事务表每次运行前为空，按邮箱的逐用例清理正常。
- **非管理员鉴权**：Run #002 的两处后端修复已生效，admin 用例稳定。
- **非 Stripe.js 加载**：在浏览器层 `route.abort()` 掉 `stripe.com` 后，仍会失败 → 排除支付步骤加载 Stripe 卡住这一条。
- **非 cold-start 首请求**：加 `retries: 2` 后同一次运行内 3 次尝试仍全部白屏。

**根因**：复用的前端 dev server（:3000 的 vite）长时间运行、经历大量 HMR 热更新后状态劣化，**间歇性地把 SPA 渲染成空白页**。特征完全吻合：同一次运行内多次重试都白屏，而换一次干净运行又全绿；与用例逻辑无关。

**修复**：重启前端 dev server（全新进程）后，本次 7/7 一次通过、0 重试。`retries`/`stripe.com` 拦截作为额外保险保留。

## 5. 建议

- 为长期稳定，建议让 Playwright 自管理一个全新的前端实例（前端 webServer 设 `reuseExistingServer: false`，并保证 :3000 空闲），避免依赖手动维护、可能劣化的常驻 dev server；或在每次跑测前重启前端。
- Stripe.js 拦截可长期保留：在全 mock 支付下，测试不应依赖外部脚本。
